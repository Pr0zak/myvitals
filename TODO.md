## #WIDGET — Phone home-screen widgets (planning)

**Plan source:** user request 2026-05-15 — "plan what widgets for the
phone would be good to have".

Pattern to mirror: `app.myvitals.fasting.FastingWidgetProvider` (already
shipped; AppWidgetProvider + RemoteViews + WorkManager 15-min refresh
+ brand vector drawables in res/xml/fasting_widget_info.xml +
res/layout/widget_fasting.xml). Each new widget = one provider +
one layout + one info xml + one tile in AndroidManifest.

Ordered by glance-value per square inch:

| Mnemonic | Subject | Size | Source | Notes |
|---|---|---|---|---|
| WIDGET-1 | Sober streak | 2×1 | `/sober/current` | Single number, increments daily; brand-green active, slate when no streak. Tap → /sober. Highest emotional ROI. |
| WIDGET-2 | Steps today + goal ring | 2×1 | `/summary/today` + `profile.extra.steps_goal` | Mirrors StepsRing from the chip grid. Updates as steps roll in via HC sync. Tap → /vitals/steps. |
| WIDGET-3 | Readiness verdict | 2×1 | `/summary/today` + `/ai/verdict/latest` | Score + 1-line AI tone if enabled. Single morning-glance summary. Tap → /. |
| WIDGET-4 | Weight to goal | 2×1 | `/ai/goals` (active weight goal) | Current/target with mini progress bar — reuses GOALS-3 payload. Tap → /weight. |
| WIDGET-5 | Workout today | 2×1 | `/workout/strength/today` | "Push day · ready" / "Push day · in progress" / "Rest day". Tap → /workout. |
| WIDGET-6 | Sleep last night | 2×1 | `/query/sleep/last` | h.m + score color. Quick "did I sleep enough". Tap → /sleep. |
| WIDGET-7 | Activity feed last | 2×2 | `/activities?limit=1` | Last ride/run with name + distance + duration + small spark of HR. Tap → activity detail. |
| WIDGET-8 | HR last-known | 2×1 | `/query/heartrate?since=…` | Most-recent BPM + relative time, pulse animation if <5min old. Tap → /heart-rate. |
| WIDGET-9 | 4×1 combo strip | 4×1 | aggregates all of WIDGET-2/3/4/5 | Single wider widget for users who don't want to manage 4 separate tiles. |
| WIDGET-10 | Configurable | 2×1 | long-press picker | One widget binary, user picks which metric it shows in widget config. Reduces clutter; doubles complexity. Defer until 3+ above ship. |

**Implementation rules** (carried over from FastingWidgetProvider):
- AppWidgetProvider + RemoteViews-only (no Compose Glance — keeps the
  binary tiny and lockscreen-compatible)
- WorkManager periodic refresh every 15 min (Android's manifest
  `updatePeriodMillis` floor is 30 min; we cheat with a Worker)
- Tap target uses the same `shortcut_route` extra pattern as
  `res/xml/shortcuts.xml`
- One small JsonCache key per widget for stale-while-revalidate so a
  home-screen-flip doesn't show a placeholder
- Brand vector drawable per widget icon, matching the existing
  `ic_shortcut_*` style

**Recommended ship order:** WIDGET-1, WIDGET-3, WIDGET-2 — the three
that satisfy the "daily ritual" loop (am I sober today? am I
recovered today? have I moved today?). The others are nice-to-haves.

## AI integration — batch 3 (deferred)

### Phone-side alert notifications

Backend produces structured alerts (`/ai/alerts`); since v0.7.128
these are generated lazily on read (when the most recent alert is
>6h old). ALERTS-1 retired the standalone web /alerts page, ALERTS-2
(v0.7.201) feeds historical alerts into Coach payloads. The companion
app still needs to surface them as system notifications.

- [ ] Add `aiAlerts()` + `markAlertsNotified()` to `BackendClient.kt`
- [ ] Inside `SyncWorker.doWork()`, after a successful sync, fetch
      `GET /ai/alerts?unacked_only=true`. For any row whose
      `phone_notified_at` is null, post a `Notifier.postAiAlert(...)`
      and `POST /ai/alerts/mark-notified` with the IDs to suppress
      duplicates.
- [ ] Extend `Notifier` with a second channel `alerts` (severity-based
      icons, distinct from the `updates` channel).
- [ ] Tapping the notification deep-links into the dashboard via
      `https://<backend>/?alert=<id>` — frontend route handler scrolls
      to the relevant card.
- [ ] Optional: a small "AI alerts" toggle in phone Settings →
      Diagnostics so users can mute the channel without touching
      Health Connect grants.

### Other AI deferrals (lower priority)

- [ ] **Diff view** on Today — "this week vs last week" side-by-side
      structured cards. Backend `POST /ai/explain/diff?ranges=…`
      returns two structured outputs and a delta narrative.
- [ ] **Source citations** — make claims in AI cards clickable to
      jump to the underlying data point ("HRV 11ms on May 6" → opens
      `/trends#hrv` zoomed to that day). Needs a structured response
      schema with `(text, link)` tuples.
- [ ] **Smart cache invalidation** — currently any tiny payload-hash
      delta re-bills. Could whitelist only material fields (changing
      `notes` shouldn't bust the sleep cache).
- [ ] **Local LLM option** — Ollama / Llama backend; same `/ai/*`
      endpoints, different SDK behind a config flag. Pure privacy
      play.
- [ ] **Voice Q&A** on the phone — TTS the answer; tap-to-speak
      input via Android speech recognizer.
- [ ] **HA env enrichment** — `env_readings` (bedroom temp/humidity)
      already in DB but not in Claude's payload. Adding it lets the
      model correlate sleep with environmental factors.
